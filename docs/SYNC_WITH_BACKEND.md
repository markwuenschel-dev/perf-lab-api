# Syncing the Frontend with the Backend API

> **The frontend API types are GENERATED from the backend's OpenAPI schema.** They
> are no longer hand-mirrored from `app/schemas/`. The old manual type-mapping
> tables and per-field ritual are retired — the workflow is now **regenerate + `tsc`**.

## How the contract flows

```
app/schemas/*.py     (Pydantic — source of truth)
        │  FastAPI → OpenAPI document
        ▼
openapi.json         (committed at the repo root — the canonical contract artifact)
        │  pnpm run gen:types      (openapi-typescript, reads ../openapi.json from web/)
        ▼
web/src/types.gen.ts (generated — never hand-edit)
        │  re-exported under friendly names + frontend-only types
        ▼
web/src/types.ts     (thin adapter — what app code imports)
```

- `openapi.json` is generated by `app/scripts/export_openapi.py` and committed. It
  is deterministic (sorted keys), so a `git diff` on it shows exactly what changed
  in the API contract.
- **`web/src/types.gen.ts`** is the verbatim openapi-typescript output. Never edit
  it by hand; it's overwritten on every `pnpm run gen:types`.
- **`web/src/types.ts`** is a small hand-curated adapter: it re-exports the
  generated `components["schemas"][...]` shapes under the friendly names the app
  already uses (`UnifiedStateVector`, `WorkoutLog`, `Modality`, …) and defines the
  few **frontend-only** types that have no backend counterpart. App code keeps
  importing from `@/types` / `../types` — only the underlying definitions changed.

## When you change an API schema

Any add/rename/remove/retype in `app/schemas/` (or a new route / response model),
both steps in the same PR:

1. **Regenerate and commit the contract:**
   ```bash
   python -m app.scripts.export_openapi      # rewrites ./openapi.json
   ```
2. **Regenerate the web types** (this repo is a monorepo — `web/` reads the
   committed `openapi.json` at the repo root, not a sibling repo):
   ```bash
   cd web
   pnpm run gen:types      # rewrites src/types.gen.ts from ../openapi.json
   pnpm run check:types    # tsc --noEmit — every break in consumers surfaces here
   ```
   Fix whatever `tsc` flags, then commit `types.gen.ts` (+ any consumer fixes).
   `pnpm run build` runs `tsc -b` too, so a stale or broken type can't ship.

That's the whole ritual: **regenerate + `tsc`**. No type table to keep in your head.

> Want to regenerate from a live API instead of the committed file? Point
> openapi-typescript at the URL instead of `../openapi.json` — but prefer the
> committed `openapi.json` for deterministic, offline builds.

## CI enforcement

`.github/workflows/ci.yml` runs `python -m app.scripts.export_openapi --check` on
every PR (Python job) and regenerates + diffs `src/types.gen.ts` (web job). If you
change a schema but forget to regenerate either artifact, the corresponding job
fails rather than letting the contract drift silently. The Python job also imports
`app.main` (boot smoke test) and runs the test suite.

## Naming notes (schema name ≠ frontend name)

A few generated schema names differ from the friendly names the web app uses; the
mapping lives in `web/src/types.ts`:

| OpenAPI schema (`components.schemas`) | Frontend alias | Route |
|---|---|---|
| `MetricsRequest` | `ComputeMetricsRequest` | `POST /compute-metrics` (legacy, no `/v1`) |
| `RegisterRequest` | (used inline) | `POST /auth/register` |

Literal/enum fields that Pydantic defines inline (e.g. `WorkoutLog.modality`) are
**not** standalone named schemas; the web derives them from the parent type
(`type Modality = NonNullable<components["schemas"]["WorkoutLog"]["modality"]>`).

## Route prefixes

- Most routers mount under `/v1`; auth is under `/auth`; the legacy field-test
  endpoint `/compute-metrics` has **no** prefix.
- The web client (`web/src/api/perfLabClient.ts`) appends `/v1` to
  `VITE_API_BASE_URL` itself, so that env var must be set **without** a `/v1` suffix.

## Frontend-only types (not in OpenAPI — keep hand-written)

These have no backend counterpart and stay defined in `web/src/types.ts`:

- `ApiError` — normalized client error shape.
- `FieldTestHandoff` — Field Test → Twin in-app handoff payload.
- `PingResponse` — defined inline in `perfLabClient.ts` (`/ping` returns a bare dict).

## Request-body ergonomics

openapi-typescript marks properties that carry a server-side **default** as
*required* in the generated types (correct for responses — the field is always
present). When **building a request body**, the backend still fills those
defaults, so you may omit them. Construct the payload with the fields the form
captures and assert the contract type, e.g.:

```ts
return { timestamp, modality, duration_minutes, session_rpe, sleep_quality,
         life_stress_inverse } satisfies Partial<WorkoutLog> as WorkoutLog;
```

`satisfies Partial<WorkoutLog>` still type-checks the fields you *do* set; the
cast covers the server-defaulted remainder. (See `buildWorkoutLog` in
`LogWorkoutModal.tsx` and `completeOnboarding` in `AuthContext.tsx`.)
