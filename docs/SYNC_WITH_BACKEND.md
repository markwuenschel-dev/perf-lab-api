# Syncing the Frontend with the Backend API

> **As of the OpenAPI type-contract migration (roadmap P1), the frontend types are
> generated from this backend's OpenAPI schema — they are no longer hand-mirrored.**
> The manual type-mapping ritual that used to live in this file is retired. What
> remains below is the small workflow that keeps the contract honest.

## How the contract flows

```
app/schemas/*.py  (Pydantic — the source of truth)
        │  FastAPI builds the OpenAPI document
        ▼
openapi.json      (committed at the repo root — the canonical contract artifact)
        │  perf-lab-web: npm run gen:types   (openapi-typescript)
        ▼
perf-lab-web/src/types.gen.ts   (generated; never hand-edited)
        │  re-exported under friendly names by
        ▼
perf-lab-web/src/types.ts       (thin adapter; frontend-only types live here too)
```

`openapi.json` is generated by `app/scripts/export_openapi.py` and committed. It is
deterministic (sorted keys), so a `git diff` on it shows exactly what changed in the
API contract.

## When you change an API schema

Any add/rename/remove/retype in `app/schemas/` (or a new route / response model):

1. **Regenerate and commit the contract:**
   ```bash
   python -m app.scripts.export_openapi      # rewrites ./openapi.json
   ```
   Commit `openapi.json` alongside the schema change in the same PR.
2. **Regenerate the web types** (in the `perf-lab-web` repo, checked out as a sibling):
   ```bash
   npm run gen:types                         # rewrites src/types.gen.ts
   npx tsc --noEmit                          # every break in consumers surfaces here
   ```
   Fix anything `tsc` flags, then commit `types.gen.ts` (+ any consumer fixes).

That's the whole ritual: **regenerate + `tsc`**. No type table to keep in your head.

## CI enforcement

`.github/workflows/ci.yml` runs `python -m app.scripts.export_openapi --check` on
every PR. If you change a schema but forget to regenerate `openapi.json`, the
**contract** job fails with “openapi.json is out of date — regenerate … and commit.”
The same workflow imports `app.main` (boot smoke test) and runs the test suite, so a
schema change that breaks the app or the contract can’t land silently.

## Naming notes (schema name ≠ frontend name)

A few generated schema names differ from the friendly names the web app uses; the
mapping lives in `perf-lab-web/src/types.ts`:

| OpenAPI schema (`components.schemas`) | Frontend alias | Route |
|---|---|---|
| `MetricsRequest` | `ComputeMetricsRequest` | `POST /compute-metrics` (legacy, no `/v1`) |
| `RegisterRequest` | (used inline) | `POST /auth/register` |

Literal/enum fields that Pydantic defines inline (e.g. `WorkoutLog.modality`) are
**not** standalone named schemas; the web derives them from the parent type
(`type Modality = NonNullable<components["schemas"]["WorkoutLog"]["modality"]>`).

## Route prefixes (unchanged)

- Most routers mount under `/v1`; auth is under `/auth`; the legacy field-test
  endpoint `/compute-metrics` has **no** prefix.
- The web client (`src/api/perfLabClient.ts`) appends `/v1` to `VITE_API_BASE_URL`
  itself, so that env var must be set **without** a `/v1` suffix.

## Frontend-only types (not in OpenAPI — keep hand-written)

These have no backend counterpart and stay defined in `perf-lab-web/src/types.ts`:

- `ApiError` — normalized client error shape.
- `FieldTestHandoff` — Field Test → Twin in-app handoff payload.
- `PingResponse` — defined inline in `perfLabClient.ts` (`/ping` returns a bare dict).
