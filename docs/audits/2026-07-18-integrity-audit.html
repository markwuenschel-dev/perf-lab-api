Executive summary

The repo is materially healthier than at the last audit. The PA-* batch closed every correctness bug the prior run found — the wellness 500-on-committed-write (C1), the benchmark rollback (C4), the bcrypt byte-limit (C13) — plus the xdist DB race, the assertion-presence gap, and the whole unwired-flag cluster. Static gates are green (ruff, pyright strict at 0 errors, 1,289 tests collectible, an 83% coverage ratchet). No category is failing and there is no known live 500-class bug.

So this run's findings are second-order, and they cluster into three shapes. Gate holes: the reader-coverage guard that would have caught a dead config flag is scoped to a single file (C16, the reason C10 survives), and the canonical domain vocabulary is hand-mirrored into a TypeScript picker with nothing binding them (C20). A CI gap: the frontend vitest suite exists but the web job never runs it — a regression in the units helpers ships green (C21). Standing decisions: five carried candidates are each a call only you can make — reconcile the model↔DB nullability/cascade drift (C12), ratchet-or-fix the cosmetic drift bucket (C7), wire-or-delete a config flag (C10) and orphan simulation stubs (C11), and decide belief idempotency on wellness re-ingest (C8).

Headline this run — C21
The frontend test suite is not wired into CI. PA-22 added vitest and a first test (web/src/lib/units.test.ts), but the web CI job runs gen:types, token checks, tsc/vite build, and lint — and never invokes vitest. A grep of .github/ for vitest/pnpm test returns zero matches. The single frontend test therefore runs only when a human runs it locally; a unit-conversion regression would merge green. This is the highest-priority quick win: one CI step, trivially red-capable.

Read-only run: the swarm inspected the repo and wrote only this report. Nothing was executed. Selection and any fix belong to a later, separately-authorized loop.
